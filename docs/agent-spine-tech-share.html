ethan@spine:~/what03

what

agent-spine = 人驾驭的自主 harness

把「长时运行 + 自主决策 + 反复 review 打磨」的编排做成纯 markdown skill，确定性动作全部委托给 npc CLI。

/spine-run

主 harness

init → plan →（每个 change）implement → review → fix → archive → 收尾，支持断点续跑

spine-coder

专职执行体

被 spawn 的 subagent，产代码 commit + summary.md，只回一行 RESULT

/spine-analyze

自迭代入口

读跨 run 指标，提 ≤3 条 harness 改进建议（只读，人审闸门）

定位：从 spec 到结果交付的 skill，不是无人值守的生产系统——人随时可介入。

ethan@spine:~/arch04

architecture

三层职责：谁思考、谁动手、谁搬砖

层角色由谁承担
智能层调度 + 决策：排计划、spawn、读一行 JSON 做分支主 session（/spine-run）
执行层写代码 + 过程日志 + 一行 RESULT 回报spine-coder subagent
底座确定性机械动作：状态 / 事件 / 模板 / 子进程 / 指标npc CLI（Python）
  • 主 session 只调度与决策——不写业务代码、不解析自然语言日志
  • 凡是能确定性完成的，一律下沉给 npc，不占智能层 context
ethan@spine:~/loop05

the loop

一个 run 的完整生命周期

npc init→ plan 拆解→ implement→ review→ fix⟲ 直到 blocking==0→ archive→ finalize
  • 自由目标（如「给认证模块加限流」）会先被自动拆解成若干 openspec change，排出 plan_order
  • 每个 change：spawn coder 实现 → npc review run 多轮独立 review → 有 blocking 就 spawn coder 修
  • 决策点（review 卡死 / archive 失败）：交互档问你，auto 档 npc auto-decide 机械判定
  • 中断后再跑 /spine-run 自动检测 needs_resume，从断点接着来

constitution

4 条核心不变量（宪法，冲突时以它为准）

01

生成 ⊥ 验证

写代码的角色，绝不评判自己的代码

02

只信结构化契约

状态以落盘数据为准，不信 LLM 散文

03

笼子 ∝ 1/(人在回路)

人驾驭 → 硬轨最小化，按需再加

04

成本分层

廉价模型只许执行，决策与验证恒 premium

来源：从生产级零信任架构（aidevos）蒸馏出的、适配「人驾驭 skill」定位的最小集。

ethan@spine:~/principles/107
01

生成 ⊥ 验证：执行者永不给自己盖章

  • coder 只生成；合格与否由独立 review（codex / Claude 引擎）判定
  • review 引擎绝不可与 coder 同源——coder 走 MiMo 时 review 必须仍是 codex/Claude
  • archive 闸门只认 review 的 blocking==0（或人类显式 override）
  • 不认 coder 自报的「我写好了」「测试全过」
npc archive run
{"ok": false, "reason": "review.blocking=2"}  # 门不开
ethan@spine:~/principles/208
02

轨迹与结构化契约是唯一真相

  • 主 session 只读 npc 返回的一行 JSON 关键字段做分支
  • 角色交接走结构化契约：coder → 一行 RESULT；npc → 一行 JSON
  • 全轨迹落 ~/task_log/<PROJ_KEY>/，跨 run 指标落 _telemetry/
  • 反模式：读 summary 原文做决策、把模板搬进 context——npc 存在就是为了消灭这个
npc review run --change add-rate-limit
{"blocking": 0, "warnings": 3, "round": 2, "stale": false}
ethan@spine:~/principles/309
03

确定性「笼子」∝ 1 / (人在回路)

  • spine 有人驾驭（人即编排者 + 验证锚）→ 笼子最小化
  • 不照搬重机器：policy 闸门、不可变 ledger、日历锁——那些是给「全程无人」准备的
  • 加任何新硬轨前先问：「这是因为去掉人了吗？」不是，就别加
  • 硬轨由 npc telemetry hotspots 指出的真实方差点位「打」出来，不预先过度设计
  • --auto 是「少打断人」的便利档，≠ 无人产品
ethan@spine:~/principles/410
04

成本分层：廉价层只许执行

  • 便宜模型（MiMo）只可用于 coder 层；决策 + review/分析恒留 premium（Claude / codex）
  • premium coder 默认走 in-session subagent（对冲 headless claude -p 的计费风险）；MiMo 恒走 headless
  • MiMo 默认不启用，按需显式开：全局 / per-phase（只把 fix 给 MiMo）/ 临时 --backend mimo
  • npc verify routing 在代码层强制：review 含 mimo = violation，mimo + in-session 亦是
npc verify routing
{"ok": true, "coder": "mimo", "review": "codex", "violations": []}
ethan@spine:~/modes11

modes

交互档 vs --auto 档

交互档（默认）

人拍板

plan 排好先给你确认；review 卡死 / archive 失败时 AskUserQuestion 让你定。适合高风险或首次上手。

--auto 档

fire-and-forget

绝不打断人：例行决策交给 npc auto-decide 机械判定，一路跑到底，只在真正卡死时才停。

  • auto 档硬规则：绝不调用 AskUserQuestion，每个分叉用确定性默认或 auto-decide
  • 每个 run 独立 worktree 隔离，ff-only 合回主干——工程目录零侵入
