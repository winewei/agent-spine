## 建议 1：把 reviewer rubric 前置进 coder 的 implement prompt，抬高首轮通过率

- **观察**：`review-r0` count=27，其中 `approve` 仅 **3**、`changes-requested` **24** → 首轮通过率 **≈11%**。该 phase retry_count_sum=27（每个 change 首轮都记一次 retry），p50=126518ms（≈2m6s）、p95=213723ms，是全表 score 最高的 hotspot。几乎每个 change 都强制进入至少一轮 fix 循环。
- **诊断**：coder（`spine-coder` subagent）在 implement 阶段没有对照 reviewer 用的同一份 focus/rubric 做自检。review focus 文本由 `npc focus` 渲染（`src/npc/focus.py`），coder 的 implement prompt 由 `npc agent`（`src/npc/agent.py`）渲染——两者是割裂的，coder 事前看不到会被 review 卡的判据。
- **建议**：在 implement prompt 渲染时，注入 review focus 的 checklist 摘要（即 reviewer 会逐条检查的 blocking 维度：validation / partial-failure / locking / test-coverage / edge-case / telemetry / concurrency），要求 coder 提交前逐条自查。**只描述，不动手改。** 落点：`src/npc/agent.py` 的 implement prompt 组装处引用 `focus` 模块的同源 rubric。
- **验证方式**：`review-r0` 的 `approve` 占比从 11% 上升；`review-r0.retry_count_sum` 与全表 `review_rounds` 总和下降。

## 建议 2：在 plan 阶段加 change 体量/复杂度门，拦住 10+ 轮的长尾

- **观察**：按 change 聚合，长尾极端：
  - `001-wire-incentive-queue-consumers`：review_rounds=**11**、blocking_total=**21**、est_tokens≈**63194**。
  - `004-m1-eval-protocol`：review_rounds=**10**、blocking_total=15、est_tokens≈**44676**。
  - 这两个 change 合计 ≈**108k** est tokens，占 30d 全量 token 的显著比例；而 27 个 change 里有 22 个 ≤3 轮就收敛。少数超大 change 吞掉了绝大部分 fix/review 成本。
- **诊断**：plan 阶段（`npc plan`，`src/npc/plan.py` 的 change 脚手架/前置门）目前不对 change 的体量或跨领域范围设限。一个 change 同时背 7 类 blocking 维度（见建议 1 的类别列表，来自当前 in-flight 的 `parallel-dag-scheduling` plan-state），review 每轮只能收敛一两类，于是轮数线性堆高。
- **建议**：在 plan 前置门加一个软性体量门——按 change 涉及文件数 / 预估 LOC / spec 中 requirement 条目数给出阈值告警，超阈值时提示拆分（或标记为 "large，允许更高 max-rounds 预算"）。**只描述，不拆分、不改代码。** 落点：`src/npc/plan.py` 前置门。
- **验证方式**：单 change 的 `review_rounds` 最大值从 11 降到个位数；`by-change` 的 token p95 下降；无新 change 落入 >8 轮长尾。

## 建议 3（减法）：`verify-tests-rerun` 笼子近 90d 零触发，候选软化/删除

- **观察**：`npc telemetry cages --since 90d` 显示 `verify-tests-rerun` 在 `runs_observed=11`（≥ 阈值 5）的窗口内触发 **0 次**，被列入 `deletion_candidates`。它是唯一 `has_data=True` 且 untriggered 的笼子（其余 12 个均为 `no_data`，事件未接线，**不可**当删除候选）。
- **诊断**：该硬轨定义于 `src/npc/telemetry.py:428`，映射到 `phase.exit` + `outcome_reason=="rerun-tests-failed"`，由 `pipeline._do_phase_exit` 在 implement / fix-rN 阶段 tests rerun 失败时 emit。11 个 run 从未命中，说明"rerun tests 失败"这条守卫路径要么场景已被上游绕开，要么是冗余守卫。
- **建议**：复核 `pipeline._do_phase_exit` 中 `rerun-tests-failed` 分支的必要性——若确认无业务场景会走到，软化或删除该守卫与对应 `_CAGE_DEFS` 条目；若它是防罕见但灾难性回归的守卫，则保留并在此备注理由。**只描述，不动手删。**
- **验证方式**：删后 `npc telemetry cages` 输出中 `verify-tests-rerun` 移出 `deletion_candidates`/`cages`，且回归测试无 tests-rerun 相关失败。
