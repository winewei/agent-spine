# 提案：`npc watch` / `npc serve` — 本机后台任务进展统一观测 — 2026-07-02

来源：kube-service-controller 工程多 session 并行时的真实痛点——一个 session 派发的后台任务
（Claude Code 原生 background sub-agent / 后台 Bash），另一个 session 与用户都无法感知进展，
只能靠 agent 主动去查。属人工提案，非 `/spine-analyze` 指标驱动。

状态：**已实施（方向调整）**——commit `9624470` 落地 `npc watch` + `npc task`，但实现路线与本提案不同：
**没有**解析 `~/.claude/projects/` 的 transcript / subagents jsonl（内部格式无兼容承诺，防御性解析成本高），
而是先落地 npc 自有的任务上报契约：后台 agent 通过 `npc task start/update/heartbeat/finish` 写
`<run_dir>/tasks/<task-id>.json`（+ `.events.jsonl`），`npc watch` 只观测这份自有契约。
取舍：观测面从"Claude Code 内部状态"收窄为"显式上报的任务"，换来零逆向工程成本与格式稳定性；
未显式上报的原生后台 sub-agent 仍不可见（本提案的原始痛点部分保留）。b) `npc serve` HTTP 服务未实施。
设计记录见 design.md §11.8。

---

## 观察

### 进展数据其实都已落盘，只是没人聚合

| 数据源 | 位置 | 覆盖什么 |
|---|---|---|
| npc run 状态 | `~/task_log/<PROJ_KEY>/<run-ts>-plan-state.json` + `active.json` + `index.jsonl` | npc pipeline 驱动的 changes 进度（implement/review/fix/archive 各 phase 状态） |
| npc 跨 run 指标流 | `~/task_log/_telemetry/events.ndjson` | 所有工程的 phase 事件、耗时、失败率 |
| Claude 主 transcript | `~/.claude/projects/<PROJ_KEY>/<session-id>.jsonl` | 每 turn 事件；`turn_duration` 事件带 `pendingBackgroundAgentCount`（session 自认还挂着几个后台 agent） |
| Sub-agent 明细 | `~/.claude/projects/<PROJ_KEY>/<session-id>/subagents/agent-*.jsonl` + `.meta.json` | 每个 sub-agent 的完整对话流；meta 含 `agentType` / `description` / `toolUseId` / `spawnDepth` |

实测样本（2026-07-02，kube-service-controller 工程 session `3f00378c`）：
transcript 尾部 `turn_duration` 事件 `pendingBackgroundAgentCount: 1`；
`subagents/` 下数十个 `agent-*.jsonl`，最新 meta 为
`{"agentType": "senior-code-developer", "description": "Write multi-cluster e2e tests and update setup scripts", ...}`。

### 关键实现约束

- **`.meta.json` 里没有 status 字段**。任务是否结束必须从对应 `agent-*.jsonl` 推断：
  mtime 新鲜度 + 尾行事件是否终态（final result / stop）。
- `~/.claude/projects` 的 jsonl 是 Claude Code 内部格式，**无兼容承诺**，解析层必须防御性
  （字段缺失降级显示，不崩）。

## 诊断：npc 目前只有"半个"这个能力

已有的：

- `npc telemetry tail/agg/hotspots` — 跨 run 事件流查询
- `npc state get` / `npc resume detect` — 单 run 状态快照

两个缺口：

1. **只见自己，不见 Claude**。npc 的事件是自己 pipeline emit 的。session 用
   `Agent(...)` 或后台 Bash 跑的任务，npc 完全不感知——而这恰恰是痛点所在。
2. **没有常驻形态**。`src/` 下无任何 serve/watch/dashboard 能力，全是一次性命令，
   pull 式快照回答不了"现在各 session 的后台任务跑到哪了"。

## 建议

给 npc 新增观测子命令，纯只读、零侵入（不写任何 session/task_log 文件），分两期：

### a) `npc watch [--project <path>] [--all]` — TUI 模式（第一期，约半天）

每 2–3 秒扫描并渲染：

- 每个 session 一节：session id、最后活跃时间、`pendingBackgroundAgentCount`
- 其下每个 sub-agent 一行：`description`、`agentType`、判活状态、最后一条动作摘要
  （从 jsonl 尾部提取最近 tool_use / assistant text 截断）
- npc run 部分直接复用 `plan-state.json`（changes × phase 进度表）
- `--all` 跨工程视图（`~/task_log/` 下已有十几个工程目录，天然支持）

判活核心逻辑（两期共用，建议独立成 `scanner` 模块）：

```
running  = agent-*.jsonl mtime < 30s ago 且尾行不是终态
finished = 尾行含最终 result（提取最后 assistant text 作结果摘要）
stale    = mtime 超阈值但无终态 → 疑似挂死，高亮
```

交叉校验：transcript 的 `pendingBackgroundAgentCount` vs 实际 running 计数不一致
→ 发现"session 认为还有后台任务但 agent 文件已停更"的挂死场景。

### b) `npc serve --port 8765` — 本机 HTTP 服务（第二期）

同一套 scanner 封成轮询/SSE 接口 + 单页 dashboard，浏览器常开 tab 看全局。
适合多工程并行场景。不做鉴权（仅 bind 127.0.0.1）。

### 实施拆分

1. `scanner` 模块：task_log 读取（复用现有契约）+ `~/.claude/projects` 防御性解析 + 判活
2. `npc watch`：TUI 渲染（终端刷新即可，不必引重依赖）
3. `npc serve`：HTTP 包装 + 单页 dashboard
4. 测试：用固化的 jsonl fixture（截真实 session 片段脱敏）锁定解析层，
   Claude Code 格式变更时 fixture 先红

### 风险与边界

- Claude Code 内部格式变更 → 解析层降级显示，fixture 测试兜底；不承诺字段级兼容
- 只读观测，不做任务控制（stop/resume 仍归各 session 自己）——控制面越界会撞权限模型
- 30s 判活阈值需可配置（长时间无输出的合法任务，如 15 分钟 e2e，避免误报 stale）
