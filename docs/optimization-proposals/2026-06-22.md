# Harness 演进提案 — 2026-06-22

来源：对照 aidevos 生产级自主系统的零信任架构（HMAC artifact 校验 + append-only ledger + 执行时再验）反推 agent-spine harness 的自报漏洞。非 `/spine-analyze` 指标驱动，属架构对标人工提案。

---

## 提案 1（第一道待加硬轨）：`npc record` 复跑测试，不裸信 RESULT 的 `tests=pass`

### 观察

`npc implement record` / `npc fix record` 当前的校验（见 `docs/cli.md §8a`、`src/agent_spine/npc/pipeline.py`）只做三件事：

- 解析 RESULT 行
- 校验 `commit != "-" && tests == "pass"`（**直接信 RESULT 行里 coder 自报的 `tests` 字段**）
- 校验 summary 文件存在 + commit 存在于 repo（`git cat-file -e`）

也就是说：**测试是否真的通过，完全采信 subagent 的自我汇报**，npc 从不实际复跑测试命令。

### 诊断

这正是 aidevos 用 HMAC + append-only ledger 要堵死的"**执行器自报权威**"漏洞。aidevos 的原则：系统**绝不信模型自报**——coder 说"改好了/测试过了"无效，必须由确定性层重算验证（`artifact-claim-precheck.mjs` 重算 `diff_sha` 验 HMAC；append-only ledger 做唯一真相源）。

agent-spine 现状是**反比例**：智能重、轨道轻。真·验证闸门是 codex review（这点对，review 是独立复核），但 `tests=pass` 是裸信的：

- coder 可能误判测试通过、跑了错误的测试命令、或测试根本没跑就填了 `pass`。
- review 阶段（codex read-only）通常不复跑测试，主要做代码评审——所以"测试假阳性"可能一路漏到 archive。

低风险场景影响有限；但一旦让 harness **无人值守跑高风险的活**（`--auto` + bypass 权限），这个自报漏洞会让"假绿"change 被 archive，且事后难追。

### 建议（只描述，不在本提案动手改）

让 `npc record` 系列在装订前**由 npc 自己复跑测试命令**，以复跑结果为准，而非 RESULT 行的 `tests` 字段：

- 测试命令来源：工程约定（如 `openspec` 的 verification surface / `pyproject` / `package.json`）或 `.npc/config.toml` 新增 `[verify] test_cmd` 配置项。
- 复跑结果 != pass → 强制 `phase exit failed` + `state set-progress status=failed reason=tests-failed-on-recheck`，**无视** RESULT 自报。
- 把复跑的退出码 / 输出尾段写进 events + telemetry（新增字段，便于 `/spine-analyze` 观测"自报与复跑不一致率"）。
- RESULT 的 `tests` 字段降级为"coder 的声明"，npc 的复跑结果为"权威判定"——二者不一致本身是高价值信号。

可选进阶（对标 aidevos）：commit 的 diff 做内容指纹，record 时校验 RESULT 声明的改动范围与实际 diff 一致，堵 commit 自报漏洞。

### 验证方式

- 加一个故意"测试失败但 RESULT 谎报 tests=pass"的用例 → record 必须判 failed。
- 跑几个真实 run 后，telemetry 里"自报 pass / 复跑 fail"的计数应可见；为 0 才说明 coder 自报可信。

---

## 演进纪律（与本提案配套）

护栏是被方差失败"打"出来的，不是预先设计的（对照 aidevos 用一年生产换来的重锁）。除"绝不信自报"这条是地基（应优先加），其余硬轨（path scope、retry policy、ledger 不可变、policy 闸门……）应**等 `npc telemetry hotspots` 指出某点位被方差咬人后再按需加**，避免一上来把 harness 锁成纯软件、让 LLM 的边际价值消失。
